# Vendoring And Package Setup

MoonBit native stubs are easiest to maintain when the C sources live in the
package directory and are generated by a deterministic script.

## Choosing A Source Strategy

Use vendored sources when:

- The package should work without system package managers.
- The upstream library is small or medium sized.
- The ABI depends on generated configuration headers.
- You need reproducible builds.

Use system linking when:

- The library is too large to vendor.
- Users are expected to install it separately.
- The ABI is stable and available on target platforms.

## Vendoring Script Checklist

Start from `templates/prepare.py` and adapt the CONFIG block for the target
library. A good vendoring script, commonly `scripts/prepare.py`, should:

- Pin upstream commit hashes or release tarballs.
- Download into an ignored cache directory such as `.prepare/`.
- Generate configured headers from upstream templates or compile definitions.
- Copy or flatten only required C/header files.
- Rewrite includes deterministically if files are flattened.
- Add a platform prelude only when needed.
- Regenerate the `native-stub` list in `moon.pkg`.
- Be idempotent: rerunning it should leave `git status --short` unchanged.

## Flattening Sources

If all stubs must live next to `moon.pkg`, flatten paths by encoding their
source path into the filename:

```text
upstream/lib/foo.c -> upstream#lib#foo.c
upstream/include/foo.h -> upstream#include#foo.h
```

Then rewrite local includes:

```c
#include "foo.h"
```

to:

```c
#include "upstream#include#foo.h"
```

Do not flatten by basename only; C projects often reuse filenames.

## Generated Headers And Type Widths

Many C libraries configure typedef widths at build time. Bindings should make
that choice explicit.

Example C wrapper checks for a library with configurable index/scalar typedefs:

```c
typedef char index_must_be_i32[
  (sizeof(library_index_t) == sizeof(int32_t)) ? 1 : -1
];
typedef char scalar_must_be_f32[
  (sizeof(library_scalar_t) == sizeof(float)) ? 1 : -1
];
```

If supporting multiple widths, expose separate MoonBit APIs or packages. Do not
silently truncate C values into `Int`.

## moon.mod.json

Native-only bindings should declare module-level target support:

```json
{
  "preferred-target": "native",
  "supported-targets": "native"
}
```

Use the smallest true target set. This prevents `moon check --target all` from
building impossible wasm/js black-box tests.
Use `templates/moon.mod.json` as the starting point.

## moon.pkg

List generated C files in `native-stub`:

```moonbit
options(
  "native-stub": [
    "wrapper.c",
    "upstream#src#foo.c",
  ],
  targets: {
    "ffi.mbt": [ "native" ],
    "README.mbt.md": [ "native" ],
  },
)
```

Every file that depends on native FFI should be gated to native targets. For a
fully native-only package this is still useful documentation.
Use `templates/moon.pkg` as the starting point; it includes markers that
`templates/prepare.py` can update.

## C Wrapper Responsibilities

The C wrapper should:

- Export only `MOONBIT_FFI_EXPORT` functions.
- Normalize ABI details into MoonBit-friendly types.
- Convert optional arrays to `NULL`.
- Copy C-allocated outputs into MoonBit-owned arrays.
- Free C allocations with the matching library free function.
- Avoid exposing raw internal C structs unless their lifetime is wrapped.

Keep the wrapper thin, but do not push unsafe ABI details into public MoonBit
code.

## Public API Boundary

The safe MoonBit layer should:

- Hide raw externs.
- Validate lengths, enum combinations, and non-empty constraints.
- Represent domain concepts with MoonBit structs.
- Reject C features that mutate hidden state or input buffers unexpectedly.
- Use `pkg.generated.mbti` to review public API changes.
